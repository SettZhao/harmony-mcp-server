# usb-serial-for-android 移植方案设计

## 1. 背景与目标

### 1.1 功能目标

将 Android 开源库 `usb-serial-for-android` 移植到 HarmonyOS API 12+ 平台，实现等价功能：
- 通过 USB 主机模式与以下芯片类型的 USB 转串口设备通信：
  - CDC/ACM（通用 USB CDC 串口，Arduino、Teensy 等）
  - FTDI FT232R/FT231X
  - Silicon Labs CP21xx（CP2102/2105/2108/2110）
  - Prolific PL2303
  - WCH CH34x（CH340/CH341）
- 支持串口参数配置（波特率、数据位、停止位、校验位）
- 支持 DTR/RTS 控制信号读写
- 支持硬件缓冲区清除

### 1.2 兼容目标

- 目标 API 版本：HarmonyOS API 12+（`runtimeOS: "HarmonyOS"`）
- 语言：ArkTS（TypeScript 严格子集）
- USB 管理 API：`@kit.BasicServicesKit` 中的 `usbManager`（`SystemCapability.USB.USBManager`）
- 权限：`ohos.permission.ACCESS_USB_DEVICE`（需要在 module.json5 中声明）

### 1.3 交付物清单

| 交付物 | 路径 |
|-------|------|
| 库核心代码 | `Template/library/src/main/ets/` |
| 库导出入口 | `Template/library/Index.ets` |

## 2. Android 库分析总结

### 2.1 架构描述

```
usb-serial-for-android
├── driver/
│   ├── UsbSerialPort (interface) — 串口核心接口
│   ├── UsbSerialDriver (interface) — 驱动接口
│   ├── CommonUsbSerialPort (abstract) — 公共基类（持有 UsbDeviceConnection）
│   ├── CdcAcmSerialDriver — CDC/ACM 驱动（含内部类 CdcAcmSerialPort）
│   ├── FtdiSerialDriver — FTDI 驱动（含内部类 FtdiSerialPort）
│   ├── Cp21xxSerialDriver — CP21xx 驱动
│   ├── ProlificSerialDriver — Prolific PL2303 驱动
│   ├── Ch34xSerialDriver — CH34x 驱动
│   ├── ProbeTable — (vendorId, productId) → Class<Driver> 映射表
│   ├── UsbSerialProber — 在 UsbManager 设备列表中探测驱动
│   └── UsbId — USB 厂商/产品 ID 常量
└── util/
    ├── SerialInputOutputManager — 异步串口 IO 循环管理
    └── HexDump — 调试用十六进制转储
```

### 2.2 可移植性结论

| 模块 | 可移植性 | 主要原因 |
|-----|---------|---------|
| `UsbId` | 直接复用（仅常量改写） | 纯常量，无平台依赖 |
| `HexDump` | 直接改写 | 纯算法，仅类型改变 |
| `UsbSerialPort`（接口） | 需适配 | 所有操作改为异步 |
| `UsbSerialDriver`（接口） | 需适配 | 类型替换 |
| `CommonUsbSerialPort` | 需适配 | 持有 `USBDevicePipe` 代替 `UsbDeviceConnection` |
| `CdcAcmSerialDriver` | 需重写（中复杂度） | 控制传输 + 批量传输全异步化 |
| `FtdiSerialDriver` | 需重写（高复杂度） | 波特率计算 + 状态字节过滤 + 全异步化 |
| `Cp21xxSerialDriver` | 需重写（中复杂度） | 控制传输 + 批量传输全异步化 |
| `ProlificSerialDriver` | 需重写（高复杂度） | 状态线程 + "黑魔法"初始化 + 全异步化 |
| `Ch34xSerialDriver` | 需重写（中复杂度） | 专有初始化序列 + 全异步化 |
| `ProbeTable` | 需重写 | 去除 Java 反射，改用工厂函数 |
| `UsbSerialProber` | 需适配 | 使用 `usbManager.getDevices()` 代替参数 |
| `SerialInputOutputManager` | 需重写 | Thread → async 循环 |

### 2.3 关键挑战

1. **同步 → 异步转换**：Android `bulkTransfer`/`controlTransfer` 是同步阻塞调用；HarmonyOS 对应 API 是 Promise 异步。整个 `read()`/`write()`/`setParameters()`/`open()`/`close()` 链路必须全部改为 `async` 方法。

2. **反射机制消除**：`ProbeTable.addDriver()` 使用 Java 反射调用 `getSupportedDevices()` 并通过 `Constructor.newInstance()` 创建实例。ArkTS 无运行时反射，改用工厂函数模式。

3. **线程模型变更**：`SerialInputOutputManager` 启动一个专用线程持续轮询。ArkTS 是单线程模型，改为 async loop（`while + await`）在 Worker 或事件循环中运行。

4. **Prolific 状态线程**：PL2303 驱动启动独立线程持续读取中断端点状态。改为在 `getStatus()` 时单次异步读取。

## 3. 整体移植方案

### 3.1 移植路径选择

**选择：纯 ArkTS 实现（不使用 Native/NAPI）**

理由：
- `@kit.BasicServicesKit` 的 `usbManager` API 已提供完整的 USB 主机控制能力（设备枚举、Interface claim、bulkTransfer、controlTransfer）
- 避免 NDK/NAPI 增加编译复杂度
- ArkTS 层完全可以实现所需逻辑

### 3.2 项目结构设计

```
Template/library/src/main/ets/
├── UsbSerialTypes.ets          # 常量（数据位、停止位、校验位、流控）
├── UsbId.ets                   # USB Vendor/Product ID 常量
├── IUsbSerialPort.ets          # IUsbSerialPort 接口定义
├── IUsbSerialDriver.ets        # IUsbSerialDriver 接口定义
├── CommonUsbSerialPort.ets     # 抽象基类
├── CdcAcmSerialDriver.ets      # CDC/ACM 驱动
├── FtdiSerialDriver.ets        # FTDI 驱动
├── Cp21xxSerialDriver.ets      # CP21xx 驱动
├── ProlificSerialDriver.ets    # Prolific 驱动
├── Ch34xSerialDriver.ets       # CH34x 驱动
```

## 4. 核心模块移植方案

### 4.1 IUsbSerialPort 接口

**Android 接口方法 → HarmonyOS 接口方法**

| Android | HarmonyOS | 备注 |
|---------|-----------|------|
| `void open(UsbDeviceConnection conn) throws IOException` | `open(pipe: usbManager.USBDevicePipe): Promise<void>` | 异步，参数替换 |
| `void close() throws IOException` | `close(): Promise<void>` | 异步 |
| `int read(byte[] dest, int timeoutMillis) throws IOException` | `read(dest: Uint8Array, timeoutMillis: number): Promise<number>` | 异步，byte[]→Uint8Array |
| `int write(byte[] src, int timeoutMillis) throws IOException` | `write(src: Uint8Array, timeoutMillis: number): Promise<number>` | 异步，byte[]→Uint8Array |
| `void setParameters(int baudRate, int dataBits, int stopBits, int parity) throws IOException` | `setParameters(baudRate: number, dataBits: number, stopBits: number, parity: number): Promise<void>` | 异步 |
| `boolean getCD() throws IOException` | `getCD(): boolean` | 同步（缓存状态），不变 |
| `void setDTR(boolean value) throws IOException` | `setDTR(value: boolean): Promise<void>` | 异步 |
| 同理 `getCTS/getDSR/getDTR/getRI/getRTS` | 同步 `boolean` | 缓存状态返回 |
| `void setRTS(boolean value) throws IOException` | `setRTS(value: boolean): Promise<void>` | 异步 |
| `boolean purgeHwBuffers(boolean flushRX, boolean flushTX) throws IOException` | `purgeHwBuffers(flushRX: boolean, flushTX: boolean): Promise<boolean>` | 异步 |

### 4.2 CommonUsbSerialPort 抽象基类

**关键字段映射**：

| Android 字段 | HarmonyOS 字段 | 说明 |
|-------------|---------------|------|
| `UsbDevice mDevice` | `protected mDevice: usbManager.USBDevice` | 设备信息 |
| `int mPortNumber` | `protected mPortNumber: number` | 端口号 |
| `UsbDeviceConnection mConnection` | `protected mPipe: usbManager.USBDevicePipe \| null` | 连接管道 |
| `byte[] mReadBuffer` | `protected mReadBuffer: Uint8Array` | 读缓冲区 |
| `byte[] mWriteBuffer` | `protected mWriteBuffer: Uint8Array` | 写缓冲区 |

**核心变换**：
- `mConnection.bulkTransfer(endpoint, buffer, len, timeout)` → `await usbManager.bulkTransfer(this.mPipe, endpoint, buffer, timeout)`
- `mConnection.controlTransfer(reqType, req, val, idx, buf, len, timeout)` → `await usbManager.usbControlTransfer(this.mPipe, {bmRequestType: reqType, bRequest: req, wValue: val, wIndex: idx, wLength: len, data: buf ?? new Uint8Array(0)}, timeout)`
- `System.arraycopy(src, srcOff, dest, destOff, len)` → `dest.set(src.subarray(srcOff, srcOff+len), destOff)`
- `synchronized(lock) { ... }` → 直接移除（ArkTS 单线程）

### 4.3 CdcAcmSerialDriver

**关键 controlTransfer 调用**：

`sendAcmControlMessage(request, value, buf)`:
```typescript
// Android: mConnection.controlTransfer(USB_RT_ACM, request, value, 0, buf, buf.length, 5000)
// HarmonyOS:
await usbManager.usbControlTransfer(this.mPipe!, {
  bmRequestType: 0x21, // USB_TYPE_CLASS | USB_RECIP_INTERFACE
  bRequest: request,
  wValue: value,
  wIndex: 0,
  wLength: buf ? buf.length : 0,
  data: buf ?? new Uint8Array(0)
}, 5000)
```

**setParameters 中的 SET_LINE_CODING**:
```typescript
// 7字节: baudRate(4 bytes LE) + stopBits + parity + dataBits
const msg = new Uint8Array(7);
msg[0] = baudRate & 0xFF;
msg[1] = (baudRate >> 8) & 0xFF;
msg[2] = (baudRate >> 16) & 0xFF;
msg[3] = (baudRate >> 24) & 0xFF;
msg[4] = stopBitsByte;
msg[5] = parityByte;
msg[6] = dataBits;
await this.sendAcmControlMessage(0x20 /*SET_LINE_CODING*/, 0, msg);
```

### 4.4 FtdiSerialDriver

**特殊处理：FTDI 2字节状态头过滤**

FTDI 设备每包数据前有 2 字节调制解调器状态字节，`filterStatusBytes()` 方法负责过滤，完整移植此方法。

**波特率计算**：`convertBaudrate()` 是纯算法函数，直接移植（`long` → `number`，`long[]` → `number[]`）。

**controlTransfer 格式**（FTDI 设备 OUT 请求）：
```typescript
// bmRequestType = USB_TYPE_VENDOR | USB_RECIP_DEVICE | USB_ENDPOINT_OUT
// = 0x40 | 0x00 | 0x00 = 0x40
```

### 4.5 Cp21xxSerialDriver

**controlTransfer 格式**：
```typescript
// REQTYPE_HOST_TO_DEVICE = 0x41 (vendor class, host-to-device, interface)
await usbManager.usbControlTransfer(this.mPipe!, {
  bmRequestType: 0x41,
  bRequest: request,
  wValue: value,
  wIndex: 0,
  wLength: 0,
  data: new Uint8Array(0)
}, USB_WRITE_TIMEOUT_MILLIS)
```

### 4.6 Ch34xSerialDriver

**波特率支持**：仅支持预定义波特率表 [2400, 4800, 9600, 19200, 38400, 115200]，其余抛出错误。

**DTR/RTS 控制**：通过 `controlOut(0xa4, ~((dtr?0x20:0)|(rts?0x40:0)), 0)` 设置握手字节。

### 4.7 ProlificSerialDriver

**简化处理**：
- 原始驱动中通过 `getRawDescriptors()` 反射检测设备类型，在 HarmonyOS 中使用 `usbManager.getRawDescriptor(pipe)` 获取原始描述符，读取第7字节（`maxPacketSize0`）判断设备类型
- 状态线程替换为单次 `bulkTransfer` 读取

### 4.8 ProbeTable

**工厂函数模式**：
```typescript
export type DriverFactory = (device: usbManager.USBDevice) => IUsbSerialDriver;

export class ProbeTable {
  private mProbeTable: Map<string, DriverFactory> = new Map();

  addProduct(vendorId: number, productId: number, factory: DriverFactory): ProbeTable {
    this.mProbeTable.set(`${vendorId}:${productId}`, factory);
    return this;
  }
  
  findDriver(vendorId: number, productId: number): DriverFactory | null {
    const key = `${vendorId}:${productId}`;
    return this.mProbeTable.has(key) ? this.mProbeTable.get(key)! : null;
  }
}
```

### 4.9 SerialInputOutputManager

**异步循环替代 Thread**：
```typescript
async start(): Promise<void> {
  this.mState = 'RUNNING';
  try {
    while (this.mState === 'RUNNING') {
      await this.step();
    }
  } catch(e) {
    this.mListener?.onRunError(e as Error);
  } finally {
    this.mState = 'STOPPED';
  }
}
```

## 5. 关键技术决策

### 5.1 同步 vs 异步 API 设计

| 选项 | 描述 | 选择 |
|-----|------|------|
| 选项A：同步包装（用 Atomics 等机制模拟同步） | 代码侵入少，但在 ArkTS 中不可行 | ❌ |
| 选项B：全部改为 async/await | 适合 HarmonyOS 编程模型，与 usbManager API 一致 | ✅ |

**决策**：选项B，所有涉及 USB 传输的方法改为 `async`，返回 `Promise<T>`。

### 5.2 反射 vs 工厂函数

| 选项 | 描述 | 选择 |
|-----|------|------|
| 选项A：运行时反射动态创建驱动实例 | ArkTS 不支持 | ❌ |
| 选项B：工厂函数 `DriverFactory = (device) => Driver` | ArkTS 完全支持 | ✅ |

**决策**：选项B。

### 5.3 USB 状态读取策略（PL2303）

| 选项 | 描述 | 选择 |
|-----|------|------|
| 选项A：持续后台线程轮询中断端点 | ArkTS 无真实多线程（Worker 过于重量） | ❌ |
| 选项B：按需单次读取（`getStatus()` 时读） | 简单可靠，状态信息少读但功能完整 | ✅ |

## 6. API 差异与兼容性说明

### 6.1 bulkTransfer 大小限制

HarmonyOS `usbManager.bulkTransfer` 文档注明：单次传输数据总量（含 pipe、endpoint、buffer 等）不超过 **200 KB**。

Android 无此限制。`CommonUsbSerialPort.DEFAULT_READ_BUFFER_SIZE` 设为 `16 * 1024`（16 KB），`DEFAULT_WRITE_BUFFER_SIZE` 设为 `16 * 1024`，均在 200 KB 限制内，**无需特殊处理**。

### 6.2 controlTransfer 格式变化

Android: `controlTransfer(requestType, request, value, index, buffer, length, timeout)`

HarmonyOS: `usbControlTransfer(pipe, {bmRequestType, bRequest, wValue, wIndex, wLength, data}, timeout)`

两者字段一一对应：`requestType=bmRequestType`, `request=bRequest`, `value=wValue`, `index=wIndex`, `length=wLength`。

### 6.3 权限声明

HarmonyOS 应用需要在 `entry/src/main/module.json5` 的 `requestPermissions` 中声明：
```json
{"name": "ohos.permission.ACCESS_USB_DEVICE"}
```

## 8. 风险评估

| 风险 | 等级 | 描述 | 应对方案 |
|-----|------|------|---------|
| 物理 USB 设备不可用 | 高 | 设备传输测试无法自动化 | 单元测试只覆盖纯逻辑，集成测试需手动验证 |
| `bulkTransfer` 200 KB 限制 | 低 | 缓冲区 16 KB，不触发限制 | 已通过参数验证 |
| ArkTS 并发读写冲突 | 低 | JS 单线程，无并发写冲突；但连续 await 顺序需保证 | 去除 synchronized，依赖事件循环顺序性 |
| Prolific 设备类型检测 | 中 | 原始 `getRawDescriptors` 反射调用已替换为 `usbManager.getRawDescriptor` | 已适配，但需实机验证 |
| FTDI 多端口设备 | 低 | 当前只支持单端口 FT232R，多端口 FT2232 需扩展 | 超出当前移植范围，已注释 |
